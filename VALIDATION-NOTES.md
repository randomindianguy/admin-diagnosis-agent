### Judge reasoning (supplementary — captured, not graded)
- **good · criterion_1 (mechanism) → yes:** The diagnosis explicitly states that Maya is a member of "data-team-ml," which is a nested subgroup under "data-team," and directly explains that access grants on the parent group don't propagate down to nested subgroups — precisely matching the criterion.
- **good · criterion_2 (reconciliation) → yes:** The diagnosis explicitly names both the group the operator checked ("data-team") and the actual subgroup Maya is a direct member of ("data-team-ml"), directly correcting the operator's claim that they checked group membership correctly.
- **wrong · criterion_1 (mechanism) → no:** The diagnosis attributes the issue to a propagation timing delay, not to Maya being in a nested subgroup that does not inherit access from the parent group. There is no mention of nested subgroup inheritance as the cause.
- **wrong · criterion_2 (reconciliation) → no:** The diagnosis does not mention any operator claim about checking group membership, nor does it identify both a group the operator checked and a separate subgroup the user is actually a direct member of. It only references "data-team" and attributes the issue to propagation delay.

## 3. Why each report is correct

**Verdict passes on both outputs** because both outputs resolve and the scenario expected resolve. The wrong output got the decision right and the reason wrong — exactly the failure mode this scenario was built to test. The ruler catches that through root_cause and the judges, not through verdict. `verdict=PASS` on the wrong output is the test passing, not the ruler missing.

**Good's root_cause** is an exact string match against the expected label. The good output was authored to pass; this field passing is expected and uninteresting.

**Wrong's root_cause fails**, and this is the central capability the chunk validates. The wrong output picks `propagation_lag` from the canonical root_cause list — confidently, fluently, from approved vocabulary — but it's not what the scenario expected. Exact-match catches it: a plausible-sounding confident wrong answer, caught structurally. That's the failure mode the architecture exists to prevent.

**Good's two judges both pass** for the reasons their prompts were written to enforce. Criterion 1 (mechanism) passes because the diagnosis names non-inheritance specifically: "access grants on a parent group don't propagate down to nested subgroups." Criterion 2 (reconciliation) passes because the diagnosis names both `data-team` (where the operator checked) and `data-team-ml` (where Maya sits), and tells the operator they checked the wrong level.

**Wrong's two judges both fail** for the same reasons, in reverse. Criterion 1 fails because the wrong output calls it a timing delay — wrong mechanism direction; non-inheritance is structural, not a clock issue. Criterion 2 fails because the wrong output agrees with the operator instead of correcting them; `data-team-ml` doesn't appear at all.

## 4. What would invalidate this ruler going forward

- **Judge prompt edits.** The current prompts are calibrated against these specific good and wrong outputs. Edit either prompt and the calibration is gone — re-run.
- **Scenario edits.** The `expected_answer` and trigger text are what the calibration is anchored to. Edits invalidate by construction.
- **Schema additions.** When Seeds 2–4 land, `GradeResult` grows (owner, expected_signal). Re-run to confirm existing fields still grade correctly and the drift guard catches any new mismatches.
- **Judge model swap.** The "clearly and specifically meets the criterion" bar is model-interpreted. Sonnet 4.6 → Opus or Haiku may tighten or loosen it. Re-validate.
- **Output sample edits.** If `good.json` or `wrong.json` change, the artifact's claim no longer holds — the ruler was validated against those specific outputs.
- **Strict-parser path armed but not exercised.** Both judges returned clean `"yes"`/`"no"` this run; normalization and fail-loud didn't fire. The first future run that triggers them should be eyeballed before trusting the ruler past chunk 1.
- **Rubric-disagreement-rate methodology not exercised at chunk-1 scale.** Calibration relies on single-labeler authoring (Sid); the course-taught <10% disagreement-rate-across-labelers methodology will need ≥10 outputs and 2–3 labelers to apply meaningfully.
- **Cross-model coverage partial** — only Sonnet↔Opus validated (below). Haiku tier remains unchecked; a judge-model swap to Haiku or a future tier should re-trigger cross-validation.
- **Verdict schema currently supports `resolve | escalate` only.** Seed 5 (out-of-scope) will require widening the verdict union to include `refuse_out_of_scope` (or equivalent). Drift guard at Q5 will catch the criteria-key side; the type union needs a separate update.
- **Cross-model judge agreement validated once against Opus 4.8** with full agreement on all four binary calls; Q3's first-line strict-parse contract held across the model swap, validating the no-sampling-params portability argument from Q6. Not re-run on every validation. Material judge-prompt edits or judge-model swap should re-trigger the cross-check.
- **System is Router-level on the course's autonomy ladder (6.1.2), not Agent-level** — no self-evaluate-and-retry cycle in the locked design. Upgrading to State Machine would require adding a cycle (e.g., retry retrieval with reformulated query on evidence-sufficiency failure); deferred to chunk 2+ as a deliberate design question, not bolted on.
- **Chunk-2 evidence-sufficiency threshold (0.21) is calibrated to `scenario.json`'s exact symptom text** (chunk-2 design decisions Q13; measured `voyage-4-lite` 2026-06-12: in_domain 0.3875, off_domain 0.0236). This is a known limitation, not a gate bug; the structural fix is chunk-7 query transformation (HyDE), which would raise the in_domain band toward the ~0.70 a mechanism-phrased query already scores and require re-deriving the threshold.
- **Chunk-2 gate override-from-resolve path is armed but unexercised** (chunk-2 design decisions Q12; confirmed live 2026-06-12). The Q12 fallback-owner + diagnosis-augmentation path fires only when the model *resolves* but a gate signal *fails*. Neither chunk-2 case reaches it: Seed 1 passes both gates (resolve stands), and the off-domain "weather" case has the model **self-escalate** (it supplied its own owner, "support-team") while sufficiency fails in parallel — so the primary is already an escalate and the fallback owner ("human-reviewer") + augmentation note never render. The path is built and type-correct; a case where the model confidently resolves on insufficient/inconsistent evidence is needed to exercise it (most likely a chunk-5+ ambiguous seed). Same armed-but-unexercised family as the notes below.
- **Chunk-2 consistency gate's catch-behavior is armed but unvalidated** (chunk-2 design decisions Q14; self-consistency, N=3, temp 0.7, measured 2026-06-12: Seed-1 → 3/3 `nested_subgroup_inheritance_gap`, pass). Only the *pass-on-a-clear-case* path is exercised — chunk 2 has no borderline/ambiguous seed, so the *catch-an-unstable-case* path (1-1-1 or adverse-2-1 → fail → Q12 override to escalate) has never fired in a real run. ### First-run finding — m04 contradicted its own intent

The first run produced 1 disagreement out of 40 cells (2.5%) on m04's c2 (ruler FAIL, intent PASS). m10 — pre-flagged as the most likely disagreement candidate — agreed cleanly; the disagreement came from an unflagged axis. Investigation showed the original m04 was a mutation-design flaw, not a ruler bug or a genuinely ambiguous text (chunk-1 retrofit design decisions Q8).

**Original m04 text (supplanted):**
> "You checked data-team, which does have the grant — but Maya isn't actually a direct member of data-team. She's in data-team-ml. This is a propagation timing delay following a recent group change: the membership update hasn't fully synced through the permission layer yet. Re-sync her account and the access should appear."

**Judge reasoning on original m04 c2 (verbatim from first run):**
> "The diagnosis names both the group the operator checked (data-team) and the actual subgroup the user is a member of (data-team-ml), but it does not explicitly correct the operator's claim... Instead, it frames the issue as a propagation/sync timing delay rather than clearly stating that the operator's check was incorrect."

**Why the judge was right:** axis 4 requires a wrong mechanism that is structurally *compatible* with the operator correction. Timing/sync framing specifically undermines correction — a sync-lag explanation implies the operator's membership check was right and access will self-heal, contradicting the just-stated "she's in the wrong group." The judge correctly read the muddied correction as failing the "clearly and specifically" bar (`judges.ts` JUDGE_SYSTEM_PROMPT).

**Resolution:** replaced (not dropped, not re-labeled). The replacement keeps the axis intent and the expected-grade vector but uses a wrong mechanism that doesn't contradict the operator correction (pending-approval workflow instead of timing delay). Full provenance and the reasoning for replacement-over-drop in chunk-1 retrofit design decisions Q8.

## 6. What would invalidate this retrofit going forward

- **Judge model swap or prompt edit.** The mutation set and the expected grades were calibrated against `claude-sonnet-4-6` reading the existing judge prompts. A model swap to Haiku or Opus, or material edits to `JUDGE_SYSTEM_PROMPT` or the per-criterion prompts in `judges.ts`, invalidates the calibration. Re-run.

- **Scenario edit.** The expected `verdict` and `root_cause` exact-match comparisons are anchored to `scenario.json`'s `expected_answer`. Any edit to those fields invalidates every mutation's verdict and root_cause expected grade. Re-run.

- **AgentOutput schema change.** If chunk 3's `refuse_out_of_scope` variant widens the AgentOutput union (per chunk-2 design decisions Q4 forward-link), the mutation files may need new envelope fields. Additive widening doesn't invalidate the chunk-1 retrofit; non-additive changes (renamed fields, changed types) do. Re-run on schema change.

- **Coverage gap surfaced by a future system output.** The seven axes plus the edge case cover the failure modes anticipated at retrofit time. A real chunk-2+ system output exhibiting a failure mode outside this coverage (e.g., a hedged-but-correct diagnosis, a verdict the gate overrode, a multi-mechanism conflation) would invalidate the claim that the grader is validated *across the relevant failure surface*. The honest response is to add the missing axis as an eleventh mutation and re-run, not to assume coverage holds.

- **Disagreement-rate methodology limit at single-labeler scale.** Per chunk-1 retrofit design decisions Q1, the retrofit substitutes ruler-grade-vs-author-intent for the course's multi-labeler triangulation. This adaptation is named, not silent — but the substitution has a structural limit: it cannot surface disagreements caused by the *author's* intent being itself miscalibrated. A second human labeler reviewing the manifest would surface that class of error. Not available at retrofit scope; flagged as a known limit.

- **Mutation-design flaws caught only after first run.** The m04 finding showed that an author can label intent correctly while constructing text that doesn't realize it. The Q8 replace-not-drop path is in scope only with a Q8-style entry naming the flaw, the structural reason, and the new design's fitness. Any future replacement without that provenance regresses to the paper-over anti-pattern.

- **Re-run cadence not specified.** This retrofit is a one-time validation, not a CI check. The runner is built to be re-run on demand (e.g., after a judge prompt edit), but no schedule is enforced. If material changes accumulate against the calibration anchors above without a re-run, the retrofit's claim quietly stales.
