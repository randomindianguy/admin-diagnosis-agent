### Judge reasoning (supplementary — captured, not graded)
- **good · criterion_1 (mechanism) → yes:** The diagnosis explicitly states that Maya is a member of "data-team-ml, which is nested under data-team," and directly explains that "access grants on a parent group don't propagate down to nested subgroups," clearly identifying that the subgroup does not inherit access from the parent group.
- **good · criterion_2 (reconciliation) → yes:** The diagnosis explicitly names both the group the operator checked ("data-team") and the actual subgroup Maya is a direct member of ("data-team-ml"), directly correcting the operator's claim about group membership verification.
- **wrong · criterion_1 (mechanism) → no:** The diagnosis attributes the issue to a propagation timing delay, not to Maya being in a nested subgroup that doesn't inherit access from the parent group. There is no mention of nested subgroup inheritance issues.
- **wrong · criterion_2 (reconciliation) → no:** The diagnosis does not mention the operator's claim about checking group membership correctly, nor does it name both the group the operator checked and the actual subgroup the user is a direct member of. It only references "data-team" and attributes the issue to a propagation timing delay.

## 3. Why each report is correct

**Verdict passes on both outputs** because both outputs resolve and the scenario expected resolve. The wrong output got the decision right and the reason wrong — exactly the failure mode this scenario was built to test. The ruler catches that through root_cause and the judges, not through verdict. `verdict=PASS` on the wrong output is the test passing, not the ruler missing.

**Good's root_cause** is an exact string match against the expected label. The good output was authored to pass; this field passing is expected and uninteresting.

**Wrong's root_cause fails**, and this is the central capability the chunk validates. The wrong output picks `propagation_lag` from the canonical root_cause list — confidently, fluently, from approved vocabulary — but it's not what the scenario expected. Exact-match catches it: a plausible-sounding confident wrong answer, caught structurally. That's the failure mode the architecture exists to prevent.

**Good's two judges both pass** for the reasons their prompts were written to enforce. Criterion 1 (mechanism) passes because the diagnosis names non-inheritance specifically: "access grants on a parent group don't propagate down to nested subgroups." Criterion 2 (reconciliation) passes because the diagnosis names both `data-team` (where the operator checked) and `data-team-ml` (where Maya sits), and tells the operator they checked the wrong level.

**Wrong's two judges both fail** for the same reasons, in reverse. Criterion 1 fails because the wrong output calls it a timing delay — wrong mechanism direction; non-inheritance is structural, not a clock issue. Criterion 2 fails because the wrong output agrees with the operator instead of correcting them; `data-team-ml` doesn't appear at all.

## 4. What would invalidate this ruler going forward

- **Judge prompt edits.** The current prompts are calibrated against these specific good and wrong outputs. Edit either prompt and the calibration is gone — re-run.
- **Scenario edits.** The `expected_answer` and trigger text are what the calibration is anchored to. Edits invalidate by construction.
- **Schema additions.** When Seeds 2–4 land, `GradeResult` grows (owner, expected_signal). Re-run to confirm existing fields still grade correctly and the drift guard catches any new mismatches.
- **Judge model swap.** The "clearly and specifically meets the criterion" bar is model-interpreted. Sonnet 4.6 → Opus or Haiku may tighten or loosen it. Re-validate.
- **Output sample edits.** If `good.json` or `wrong.json` change, the artifact's claim no longer holds — the ruler was validated against those specific outputs.
- **Strict-parser path armed but not exercised.** Both judges returned clean `"yes"`/`"no"` this run; normalization and fail-loud didn't fire. The first future run that triggers them should be eyeballed before trusting the ruler past chunk 1.
- **Rubric-disagreement-rate methodology not exercised at chunk-1 scale.** Calibration relies on single-labeler authoring (Sid); the course-taught <10% disagreement-rate-across-labelers methodology will need ≥10 outputs and 2–3 labelers to apply meaningfully.
- **Cross-model coverage partial** — only Sonnet↔Opus validated (below). Haiku tier remains unchecked; a judge-model swap to Haiku or a future tier should re-trigger cross-validation.
- **Verdict schema currently supports `resolve | escalate` only.** Seed 5 (out-of-scope) will require widening the verdict union to include `refuse_out_of_scope` (or equivalent). Drift guard at Q5 will catch the criteria-key side; the type union needs a separate update.
- **Cross-model judge agreement validated once against Opus 4.8** with full agreement on all four binary calls; Q3's first-line strict-parse contract held across the model swap, validating the no-sampling-params portability argument from Q6. Not re-run on every validation. Material judge-prompt edits or judge-model swap should re-trigger the cross-check.
- **System is Router-level on the course's autonomy ladder (6.1.2), not Agent-level** — no self-evaluate-and-retry cycle in the locked design. Upgrading to State Machine would require adding a cycle (e.g., retry retrieval with reformulated query on evidence-sufficiency failure); deferred to chunk 2+ as a deliberate design question, not bolted on.
- **Chunk-2 evidence-sufficiency threshold (0.21) is calibrated to `scenario.json`'s exact symptom text** (CHUNK2-DESIGN-DECISIONS Q13; measured `voyage-4-lite` 2026-06-12: in_domain 0.3875, off_domain 0.0236). The V4 chip auto-populates that exact string, so the canonical demo path is safe — but **free-text input variants of Seed 1 (paraphrases) may score below 0.21 and escalate**, because raw symptoms sit low against the mechanism-language runbook (symptom↔mechanism vocabulary distance, course ref 8.2.6). This is a known limitation, not a gate bug; the structural fix is chunk-7 query transformation (HyDE), which would raise the in_domain band toward the ~0.70 a mechanism-phrased query already scores and require re-deriving the threshold.
- **Chunk-2 gate override-from-resolve path is armed but unexercised** (CHUNK2-DESIGN-DECISIONS Q12; confirmed live 2026-06-12). The Q12 fallback-owner + diagnosis-augmentation path fires only when the model *resolves* but a gate signal *fails*. Neither chunk-2 case reaches it: Seed 1 passes both gates (resolve stands), and the off-domain "weather" case has the model **self-escalate** (it supplied its own owner, "support-team") while sufficiency fails in parallel — so the primary is already an escalate and the fallback owner ("human-reviewer") + augmentation note never render. The path is built and type-correct; a case where the model confidently resolves on insufficient/inconsistent evidence is needed to exercise it (most likely a chunk-5+ ambiguous seed). Same armed-but-unexercised family as the notes below.
- **Chunk-2 consistency gate's catch-behavior is armed but unvalidated** (CHUNK2-DESIGN-DECISIONS Q14; self-consistency, N=3, temp 0.7, measured 2026-06-12: Seed-1 → 3/3 `nested_subgroup_inheritance_gap`, pass). Only the *pass-on-a-clear-case* path is exercised — chunk 2 has no borderline/ambiguous seed, so the *catch-an-unstable-case* path (1-1-1 or adverse-2-1 → fail → Q12 override to escalate) has never fired in a real run. Same shape as the strict-parser caveat above: the mechanism is built and type-correct, but its triggering case awaits a chunk-5+ seed with genuine ambiguity, at which point N and temperature may need tuning. The technique is standard-literature (Wang et al. 2022), not from the course.
